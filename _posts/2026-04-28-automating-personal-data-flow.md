## 서론: 왜 개인정보 처리 흐름도(DFD)는 항상 구닥다리일까?

보안 및 개인정보보호 담당자라면 공감하시겠지만, **개인정보 처리 흐름도(Data Flow Diagram)**를 현행화하는 것은 고역에 가깝습니다. 서비스는 매일 변하고, 새로운 API가 생겨나는데, 정작 흐름도는 작년 인증 심사 때 그려둔 파워포인트 파일에 멈춰있는 경우가 허다합니다.

이러한 **'현행화의 공백'**은 단순한 귀찮음을 넘어 심각한 컴플라이언스 리스크를 초래합니다. 누락된 국외 이전 노드나 식별되지 않은 수탁사는 곧바로 과태료와 직결될 수 있기 때문입니다. 

그래서 저는 결심했습니다. **"흐름도도 코드(Code)처럼 관리하고 자동으로 그리자!"**

---

## 1. 핵심 아이디어: JSON을 입력으로, Mermaid를 출력으로

흐름도를 자동화하기 위해 선택한 방식은 **구조화된 데이터(JSON)로부터 다이어그램(Mermaid.js)을 생성**하는 것이었습니다.

### 입력 데이터 예시
```json
{
  "service": "Order System",
  "nodes": [
    { "id": "User", "label": "이용자", "type": "subject" },
    { "id": "Web", "label": "프론트엔드", "type": "system" },
    { "id": "DB", "label": "고객 DB", "type": "storage", "critical": true }
  ],
  "flows": [
    { "from": "User", "to": "Web", "data": ["성명", "전화번호"], "action": "주문 입력" },
    { "from": "Web", "to": "DB", "data": ["성명", "전화번호"], "action": "저장" }
  ]
}
```

이러한 형식을 사용하면 개발팀의 API 명세서나 DB 스키마 정보를 파싱하여 자동으로 입력값을 생성할 수 있습니다.

## 2. 주요 구현 기능

### 🛠️ 위험 노드 자동 식별
가장 심혈을 기울인 부분은 **법적 위험 노드 시각화**입니다. Python 스크립트가 데이터 흐름을 분석하여 다음과 같은 경우 Mermaid 문법에 특별한 스타일(예: 붉은색 테두리, 경고 아이콘)을 추가합니다.
- **국외 이전**: 데이터가 해외 리전이나 해외 서버로 전송되는 경우
- **제3자 제공**: 외부 제3자 서비스로 개인정보가 전달되는 경우
- **고위험 정보**: 주민등록번호 등 고유식별정보가 암호화되지 않은 구간을 통과하는 경우

### 📊 실시간 렌더링 (Flask + Mermaid.js)
사용자가 웹 대시보드에서 JSON을 수정하면 즉시 우측 화면에 흐름도가 갱신되도록 구현했습니다. 이는 Mermaid.js의 클라이언트 사이드 렌더링 기능을 활용한 것으로, 별도의 이미지 생성 서버 없이도 빠르고 가벼운 인터페이스를 유지할 수 있었습니다.

## 3. 결과 및 실무 적용 효과

**Personal Data Flow Mapper**를 도입한 후, 저희 팀에는 다음과 같은 변화가 생겼습니다.

1. **작업 시간 80% 단축**: 파워포인트로 노드를 하나하나 그리던 시간이 사라졌습니다. 이제는 JSON 데이터만 업데이트하면 끝입니다.
2. **증적의 정확성 확보**: "데이터는 흐르는데 흐름도에는 없다"는 심사원 지적이 사라졌습니다. 시스템 데이터와 흐름도가 동기화되기 때문입니다.
3. **커뮤니케이션 도구로의 활용**: 개발팀과 보안팀이 동일한 '데이터 흐름'을 보며 리스크를 논의할 수 있게 되었습니다.

## 결론: 코드로 관리하는 프라이버시(Privacy as Code)
